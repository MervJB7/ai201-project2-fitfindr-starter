# FitFindr

FitFindr is a multi-tool AI agent that helps users find secondhand clothing pieces and generates outfit ideas based on their existing wardrobe.

## Tool Inventory

### 1. `search_listings`
* **Purpose:** Searches the mock listings dataset using keywords, size, and price filters, returning matching items sorted by keyword relevance.
* **Inputs:** * `description` (str): Keywords describing the desired item.
  * `size` (str | None): Size filter (optional).
  * `max_price` (float | None): Maximum price filter (optional).
* **Outputs:** `list[dict]` containing the matching listing dictionaries.

### 2. `suggest_outfit`
* **Purpose:** Uses an LLM to suggest 1-2 outfit combinations combining a newly found thrift item with pieces from the user's existing wardrobe.
* **Inputs:**
  * `new_item` (dict): The selected listing dictionary from `search_listings`.
  * `wardrobe` (dict): The user's wardrobe data.
* **Outputs:** `str` containing the LLM-generated outfit suggestions.

### 3. `create_fit_card`
* **Purpose:** Generates a short, casual, Instagram-style caption for the complete outfit, mentioning the item's name, price, and platform.
* **Inputs:**
  * `outfit` (str): The outfit suggestion string generated by `suggest_outfit`.
  * `new_item` (dict): The selected listing dictionary.
* **Outputs:** `str` containing the generated caption.

---

## Planning Loop

The agent utilizes a sequential, conditional planning loop defined in `run_agent()`:

1. **Parse:** An LLM extracts the `description`, `size`, and `max_price` from the user's natural language query.
2. **Search:** The agent calls `search_listings`. 
   * *Condition:* If the results list is empty, the loop terminates early, sets an error message, and returns the session. 
   * *Condition:* If results are found, the agent selects the top result (`results[0]`) and proceeds.
3. **Suggest:** The agent calls `suggest_outfit` using the selected item and the user's wardrobe.
4. **Generate:** The agent calls `create_fit_card` using the outfit suggestion and the selected item.
5. **Return:** The completed session dictionary is returned to the Gradio interface.

---

## State Management

State is managed via a single `session` dictionary that is passed through the planning loop. No tool receives the full session dictionary directly; the agent extracts and passes only the specific parameters each tool requires. 

The session tracks:
* `query`: The original user input.
* `parsed`: The extracted search parameters.
* `search_results`: The full list of matches.
* `selected_item`: The top match (passed to subsequent tools).
* `wardrobe`: The user's wardrobe.
* `outfit_suggestion`: The output from Tool 2 (passed to Tool 3).
* `fit_card`: The final output from Tool 3.
* `error`: Populated only if a tool fails or terminates the loop early.

---

## Error Handling

Every tool is designed to handle its own failure modes gracefully without crashing the application:

* **`search_listings`:** 
If no items match the parsed filters, it returns an empty list `[]`. The planning loop detects this, sets `session["error"]` to a helpful message ("No listings found for your search. Try different keywords, a larger size, or a higher price."), and halts execution so subsequent tools are not called with null data.
* **`suggest_outfit`:** 
If the user's wardrobe is empty (`wardrobe["items"]` is empty), it does not crash. Instead, it dynamically switches its LLM prompt to provide general styling advice for the thrifted piece rather than wardrobe-specific combinations.
* **`create_fit_card`:** 
Contains a guardrail against an empty `outfit` string. If triggered, it returns the string `"Unable to generate fit card — no outfit suggestion was provided."` bypassing the LLM call entirely.

## Spec Reflection

* **How the spec helped:** 
Writing the planning document beforehand forced me to define the exact data structures and failure conditions for each tool. This made prompting the LLM for code generation much easier because the inputs, outputs, and edge cases were already decided.
* **Where the implementation diverged:** 
Initially, the spec only loosely defined how the user query would be parsed into filters. In the actual implementation of `agent.py`, I had to add an explicit LLM call at the very beginning of the loop to parse the natural language query into the strict `description`, `size`, and `max_price` variables required by `search_listings`.

## AI Usage

* **Instance 1:** 
I gave the AI my Tool 2 (`suggest_outfit`) spec from `planning.md` and asked it to implement the function using the Groq API. It generated the correct API call structure, but I had to manually adjust the prompt string inside the code to ensure the LLM output was formatted naturally instead of outputting markdown lists.
* **Instance 2:** 
I provided the AI with my architecture diagram and state management plan to generate the `run_agent()` planning loop. It successfully wired the tools together, but I had to override and rewrite the query parsing logic in Step 1 to ensure it safely handled `None` values and typecast the `max_price` to a float.
